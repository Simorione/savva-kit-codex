## Метод (читать первым)

Проект построен по **4-слойной модели** (Карпатый + Insomnia). Подробно — `method/`:

```
raw/        ← immutable источники. Не редактируем, только добавляем.
inbox/      ← буфер классификации. Всё новое падает сюда до сортировки.
workspace/  ← черновики, эксперименты. TTL 14 дней.
wiki/       ← SSOT (единый источник правды). Версионируемый markdown с YAML-фронтматтером.
```

**Главный инвариант:** в `wiki/` пишет только роль `wiki-maintainer`. Никаких прямых правок по дороге.

## Нерушимые правила

1. **Подтверждение перед действием.** Есть явный глагол-указание («сделай», «добавь») —
   выполняй; нет — сначала опиши план текстом и спроси «делаем?». (`method/06-confirm-before-act.md`)
2. **Ничего никуда не пушить/публиковать** без явного разрешения владельца на конкретное действие.
3. **Секреты** (ключи, пароли, доступы) никогда не пишутся в файлы/вики/логи. Только в окружение на лету.
4. **Не плоди сущности.** Перед созданием новой entity/поля — сверься с моделью данных: нет ли
   уже подходящего? Переиспользуй. (`method/04-contract-first.md`)
5. **Крупная/денежная фича не закрыта**, пока не пройден контур проверки — и проверяет НЕ тот,
   кто писал. (`method/03-review-contour.md` + `method/05-gradation.md`)
6. **Реальность из вики, не из головы.** Числа, реквизиты, параметры — из `wiki/` или спросить.

## Как работать

- Получив задачу — сначала **перескажи своими словами**, как понял, и задай вопросы по
  неоднозначным пунктам (владелец мог ошибиться/забыть деталь). Не неси сразу реализацию.
- Итеративно: каркас решения → согласование → детали. Не вываливай гигантский дифф без сверки.
- Тяжёлую проверку (санитар + аудитор безопасности) гоняй на деньги/оплату/доступы/роли/схему БД;
  мелочь и чистый фронт — зелёной сборкой и взглядом владельца.
