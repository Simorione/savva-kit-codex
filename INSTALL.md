# Установка и подключение

Набор не требует сборки. Нужно только разложить скелет проекта и «скормить» роли своему
ИИ-ассистенту. Два пути ниже независимы — выбери тот, что под твой инструмент.

---

## Путь 1. Git-репозиторий (Codex, Cursor, Claude Code)

### 1.1 Разложить скелет 4-слойки в свой проект

```bash
./bootstrap.sh /путь/к/твоему/проекту
```

Скрипт:
- создаст `raw/ inbox/ workspace/ wiki/` внутри твоего проекта (если их нет);
- положит `wiki/meta/` (frontmatter-schema, propagation-rules) и стартовый `wiki/index.md`;
- скопирует контракт `CLAUDE.md`/`AGENTS.md` в корень проекта;
- предложит проставить плейсхолдеры (`{{PROJECT}}`, `{{OWNER}}` и т.п.).

### 1.2 Подключить роли

`bootstrap.sh` кладёт метод, роли и хардгейты в `.savva-kit/` целевого проекта, а слои
`raw/ inbox/ workspace/ wiki/` — в корень; ссылки в разложенном `AGENTS.md`/`CLAUDE.md`
переписываются на `.savva-kit/...` автоматически.

- **Codex** читает `AGENTS.md` в корне автоматически — это и есть точка входа (роль оркестратора).
  **Важно про границу:** файлы в `.savva-kit/agents/` — это **роли-промпты, которые оркестратор читает по
  твоему указанию**, а НЕ нативно зарегистрированные Codex-субагенты. Codex не подхватывает их
  как именованных агентов сам: для этого нужен отдельный слой `.codex/agents/*.toml` (`name`,
  `description`, `developer_instructions`) — в набор он пока не входит. Практически это значит:
  «позови санитара» работает так, что оркестратор открывает `.savva-kit/agents/sanitar.md` и играет роль по
  нему, отдавая на вход голый дифф (см. контур проверки). Для настоящей изоляции ролей друг от
  друга (чтобы у проверяющего не было контекста автора) заводи их как отдельные чаты/агенты.
- **Claude Code**: используй родную версию набора (`savva-kit-claude`) — там роли уже оформлены
  как субагенты `.claude/agents/*.md` с YAML-шапками.
- **Cursor**: вынеси правила в `.cursorrules` (склей `AGENTS.md` + нужные `.savva-kit/method/*`).

### 1.3 Первый запуск

Скажи ассистенту: **«Прочитай контракт проекта и метод в `.savva-kit/method/`, представься по роли
оркестратора, кратко перескажи правила»**. Дальше работай задачами.

## Путь 2. ChatGPT: Custom GPT или Проект

ChatGPT не исполняет файлы-агенты автоматически — их нужно вложить руками.

### 2.1 Главный агент (оркестратор)

1. Создай **Custom GPT** (или ChatGPT Project).
2. В поле **Instructions** (System prompt) вставь содержимое
   `prompts-chatgpt/00-orchestrator.system.md`.
3. В **Knowledge** загрузи файлы (из клонированного репозитория набора): все `method/*.md` +
   `agents/*.md` + `example-izolation/ERP-CLAUDE.md`.
4. Проставь плейсхолдеры прямо в Instructions под свой проект.

### 2.2 Роли-специалисты

Контур проверки требует, чтобы фичу проверял **НЕ тот, кто писал**. В ChatGPT это = **отдельный
чат/GPT под каждую read-only роль**, чтобы у проверяющего не было контекста автора:

- `prompts-chatgpt/sanitar.system.md` → отдельный GPT «Санитар»
- `prompts-chatgpt/auditor-sec.system.md` → отдельный GPT «Аудитор безопасности»
- `prompts-chatgpt/revizor.system.md` → отдельный GPT «Ревизор»

Передаёшь им дифф/описание фичи как ввод — получаешь список находок.

### 2.3 Рабочий цикл в ChatGPT

1. Ставишь задачу оркестратору → он расшифровывает и задаёт вопросы.
2. Реализация (сам или через роль-инженера в отдельном чате).
3. Для крупной/денежной фичи — прогоняешь через санитара / аудитора / ревизора (отдельные GPT).
4. Находки сводишь у оркестратора → решаешь «чинить сейчас / в долг / оставить».
5. Итог канонизируешь в `wiki/` (роль wiki-maintainer).

## Граф кода (опционально, но сильно помогает)

Роли `engineer-architect`, `sanitar`, `revizor` умеют опираться на граф кода для навигации
вместо слепого перечитывания файлов. Инструмент внешний — см. `.savva-kit/skills/graphify/README.md`.
Без него роли работают, просто чаще читают исходники напрямую.
