# ПРИМЕР: контракт пространства ERP (реальный проект, секреты сняты)

> Это **живой worked-example** — как весь метод выглядит на реальной ERP/CRM-системе
> квест-центра «Изоляция», из практики которого собран этот набор. Оставлен намеренно
> конкретным (имена, стек, фичи), чтобы было видно, как абстрактные правила ложатся на
> реальность. **Все секреты вырезаны** (IP стенда, пароли, хосты подрядчика, email заменены
> на `<плейсхолдеры>`). Свой контракт делай из `templates/project-skeleton/`, а этот читай
> как образец.

---

## 1. О пространстве

**Что это:** изолированное инженерное пространство для собственной разработки и развития
ERP/CRM-системы «Изоляции» (booking, CRM, финансы, персонал, аналитика). Здесь: исходный код,
карта архитектуры, технические рецепты, граф кода, развёртывание, бэкапы, профильные роли.

**Зачем отдельно от маркетинговой вики:** разные домены (там маркетинг, здесь код), изоляция
контекста (вход «в режиме инженера» без маркетингового шума → меньше ошибок), не засорять одно
другим.

**Идентичность:** основная рабочая роль пространства — `engineer-architect`. Оркестратор
(«Искин2») остаётся продюсером сверху: ставит задачи, принимает результат.

**Цель одной фразой:** _довести ERP до состояния, когда мы сами развиваем её быстрее и дешевле
подрядчика, по лекалам лучших на рынке._

---

## 2. Граница ответственности и доступы

| Ресурс | Право |
|---|---|
| это пространство (`ERP/**`) | полное чтение/запись (кроме `wiki/` — п.4) |
| маркетинговая вика проекта | **только чтение** (реквизиты, операционка — подтягивать, не дублировать) |
| репозитории внешнего подрядчика (`<vendor-gitlab>`) | **ТОЛЬКО ЧТЕНИЕ. Никогда не пушить, не менять, не создавать MR.** |
| боевой ERP (`<prod-erp-host>`) | только чтение |
| всё вне рабочего пространства | запрещено без явного разрешения владельца |

**🔒 Правило №1 (нерушимое):** ничего никуда не пушить и не публиковать без явного разрешения
владельца на конкретное действие. Любой remote (подрядчик, GitHub, сервер) — по умолчанию
`git push`/MR/публикация НЕ выполняются никогда. Рабочие копии создаются со **снятым remote**
(`git remote remove origin`), чтобы push был физически невозможен.

**🔒 Секреты:** учётные данные подрядчика / боевого ERP / БД **никогда не записываются** в
файлы, вику, память, бэкапы. Только в окружение на лету.

## 3. Структура (фрактал 4-слойки)

```
ERP/
├── CLAUDE.md              ← этот файл (контракт)
├── .claude/
│   ├── agents/            ← engineer-architect, frontend-dev, sanitar, auditor-sec, revizor
│   └── skills/            ← graphify и др. под стек
├── raw/
│   ├── source/            ← зеркала репозиториев (backend/frontend/widget) — НЕ редактировать
│   └── access.md          ← как получать доступ (БЕЗ секретов)
├── inbox/                 ← буфер классификации
├── workspace/
│   ├── deploy/            ← docker-compose окружение для локального запуска
│   ├── dev/{backend,frontend}  ← рабочие копии (remote снят)
│   └── graphify-out/      ← граф кода
├── wiki/                  ← SSOT (пишет только wiki-maintainer)
│   ├── index.md
│   ├── architecture/      ← карта модулей «где что лежит»
│   ├── data-model/        ← 42 entity и связи
│   ├── api/               ← карта REST-контроллеров
│   ├── recipes/           ← технические рецепты (баг → решение)
│   └── decisions/         ← лог технических решений
└── backups/               ← снапшоты кода и БД
```

**Слой `raw/source/` — неизменяемый.** Эксперименты и правки — только на рабочих копиях в
`workspace/`.

## 4. Правила работы (ядро)

- **Вика через wiki-maintainer.** В `wiki/` пишет только он. Черновики — в `workspace/`.
- **Подтверждение перед действием.** Есть явный глагол — выполняем; нет — план и «делаем?».
- **Не принимать частичное «да» за полное согласие.** Ответ на один пункт составного вопроса
  не распространяется на соседние; идея от ИИ требует отдельного подтверждения; перед
  делегированием роли — сверить КАЖДЫЙ пункт ТЗ с явным указанием владельца.
- **Перед экспериментом с кодом/БД — бэкап** в `backups/` и запись в `wiki/decisions/`.
- **Развёртывание вне рабочего пространства** (Docker/порты/systemd на сервере) — спрашивать явно.
- **Реальность из вики, не из головы.**
- **Расшифровка перед выполнением.** Получив задачу — не бросаться делать: пересказать своими
  словами, задать вопросы по важным пунктам (владелец мог ошибиться/забыть).

- **Contract-first для сквозных фич (анти-костыль, анти-дубль).** Любая фича, затрагивающая И
  фронт, И бэкенд, начинается с согласованного контракта (поля/эндпоинты) на базе существующей
  модели. Предохранители: (1) перед новой сущностью — сверка с data-model, переиспользовать
  (как оплата через готовый `PaymentMethod.CERTIFICATE`, клиент через `AuthUser`); (2)
  фронтендер не выдумывает бэкенд-поля — эскалирует оркестратору; (3) оркестратор — интегратор,
  ловит дубли/костыли; (4) семантический долг фиксируется в бэклоге, не зарывается молча (как
  вынос кошелька из FIXED-хака в `PromocodeType.WALLET`).

- **Конвейер проверки после крупной/денежной фичи.** Фича не закрыта, пока не пройден контур —
  и проверяет НЕ тот, кто писал: (1) метрики → (2) sanitar (экология) → (3) auditor-sec
  (безопасность/деньги, «проверено X / не покрыто Y») → (4) revizor (связность) → (5) гейты
  закрытия (зелёная сборка; обратимость миграций; мониторинг ошибок) → (6) только потом
  wiki-maintainer канонизирует.

- **Градация дотошности:** 🔴 деньги/оплата/доступы/роли/схема БД → полный контур ВСЕГДА (даже
  если правка чисто во фронте); 🟡 средняя бэк-фича без денег → сборка + revizor + быстрый взгляд
  санитара; 🟢 мелочь/косметика/чистый фронт → зелёная сборка + взгляд владельца.
